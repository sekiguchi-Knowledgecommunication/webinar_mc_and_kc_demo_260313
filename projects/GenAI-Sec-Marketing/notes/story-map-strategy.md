# ストーリーマップ戦略分析: GenAI セキュリティガードレール設計サービス

> **バージョン**: 1.0.0 | **作成日**: 2026-02-22 | **分類**: サービス展開戦略

---

## 1. 戦略コンテキスト

### 1.1 現在地

ナレコム（ナレッジコミュニケーション）は **日本初の生成AIセキュリティコンサルティング** として、GenAI セキュリティガードレール設計サービスのプレローンチ段階にある。最優先目標は **ファーストカスタマーの獲得と事例化** である。

### 1.2 サービスの独自性

| 差別化要素 | 内容 |
|:---|:---|
| **4プラットフォーム統合** | Databricks / F5・Cisco / Microsoft AI Foundry の統合ガードレール |
| **ISO/IEC 42001 準拠** | AI マネジメントシステム規格への準拠を商談初期から織り込む |
| **多層防御** | 推論時リアルタイム防御 + 継続的レッドチーミング + 監査証跡の3層 |
| **構造化アプローチ** | 商談初期から PoC 成功条件を定量定義し、意思決定を加速する |

### 1.3 マーケット仮説

- **需要**: 生成AI導入が加速する中、セキュリティ・ガバナンスが追いついていない企業が多数存在
- **ペイン**: 専門人材不足、急速な技術進化、チーム間連携不足、高コスト構造の4要因
- **機会**: 個別製品の導入支援はあるが、E2E でガバナンスを設計する統合サービスは日本市場に不在

## 2. ペルソナ分析

### 2.1 ペルソナ・マトリクス

顧客側に4つのペルソナが存在し、商談フェーズごとに関与度が変化する。

| ペルソナ | 象限 | 商談初期 | 技術WS | 提案・計画 |
|:---|:---|:---|:---|:---|
| **P-001: CISO** | 調達者・購入者 | **高** | 中 | **高** |
| **P-002: AI基盤TL** | 影響者 | 中 | **高** | **高** |
| **P-003: AIアプリPJリーダー** | エンドユーザー | 低 | **高** | 中 |
| **P-004: 経営層（CIO/CDO）** | 内部SH | **高** | 低 | **高** |

### 2.2 Primary Persona の選定

**Primary Persona: P-001（CISO / セキュリティ責任者）**

理由:
1. **最終意思決定者かつ予算承認者** -- 事例化には CISO の Go サインが不可欠
2. **全フェーズに関与** -- 初回ヒアリングから Go/No-Go 判定まで一貫して関わる
3. **最大のペインポイント保有者** -- AI セキュリティの責任を負いながら専門人材・知見が不足

### 2.3 ペルソナごとの購買ジャーニー

```
P-004（経営層）: 投資判断 → ビジョン合意 ──────────────→ ROI確認 → Go/No-Go
P-001（CISO）  : 課題共有 → スコープ合意 → 成功条件定義 → 監査設計 → 予算承認
P-002（AI基盤）:           → 技術評価  → 設計合意  → PoC実施  → 体制合意
P-003（AIアプリ）:          → ドメイン要件 → 受入テスト → 運用要件確認
```

## 3. サービス展開フェーズの定義

### 3.1 成長ロードマップ

```
R1（MVP: 事例化）       R2（拡張: 複数顧客）     R3（スケール: マーケットフィット）
━━━━━━━━━━━━━━━━━━━    ━━━━━━━━━━━━━━━━━━━━━━   ━━━━━━━━━━━━━━━━━━━━━━━━━━
ファーストカスタマー     3〜5社の横展開          パッケージ化・パートナー展開
1社で PoC → 本番移行    パターンの型化         自動化・セルフサービス化
最小限の商談ツール       提案テンプレート整備     マーケティング本格化
属人的だが深い           再現可能な手法         スケーラブルなオペレーション
```

### 3.2 R1（MVP）の設計思想

**「最小で最大効果」の原則**:

1. **チェックリスト駆動の商談**: guardrail-design-checklist.md を武器に、構造化された商談プロセスで「この会社は分かっている」感を演出
2. **Walking Skeleton で PoC を確実に成功させる**: 全フィーチャーを薄くカバーし、E2E の価値を証明
3. **監査証跡で差別化を決定づける**: 競合が手薄な「ガバナンス・監査」領域で圧倒的優位を確保
4. **事例化のための証跡確保**: PoC の結果を構造化し、次の商談で活用可能なアセットにする

### 3.3 リリースごとの投資対効果

| リリース | 投資 | 期待効果 | 成功判定 |
|:---|:---|:---|:---|
| **R1** | 1社分の PoC 工数 + 商談ツール整備 | ファーストカスタマー獲得、事例公開許可 | 本番移行の契約締結 |
| **R2** | 提案テンプレート + パターン型化 | 3〜5社の受注、商談リードタイム短縮 | 月次の新規受注 |
| **R3** | 自動化投資 + パートナー体制 | マーケットリーダーポジション確立 | 業界認知度・指名受注 |

## 4. フィーチャーと商談フェーズの対応分析

### 4.1 フィーチャーの時系列マッピング

チェックリストの FT-01〜FT-07 は、そのまま **商談フェーズの時系列** に対応する。

```
商談開始 ─────────────────────────────────────────────────→ 受注・PoC開始
  │                                                           │
  FT-01        FT-02        FT-03        FT-04/05      FT-06        FT-07
  ビジョン・    スコープ      成功条件      ガードレール/   ガバナンス・   ロードマップ
  課題合意      定義         定義         レッドチーム設計  監査設計       ・体制
  │            │            │            │              │            │
  初回ヒアリング ──→ 技術WS ────────────→ 提案書策定 ──→ 計画確定
```

### 4.2 各フィーチャーのビジネス価値

| FT | 商談での価値 | 事例化への貢献 |
|:---|:---|:---|
| **FT-01** | 初回で「分かっている」感を演出 → 競合排除 | 顧客の生の課題を事例に |
| **FT-02** | スコープ合意で手戻り防止 → 受注確度向上 | スコープ定義の型を資産化 |
| **FT-03** | 成功条件の先行合意 → 意思決定迅速化 | KPI 実績を事例の定量証拠に |
| **FT-04** | 技術的信頼の獲得 → 「守れる」確信 | アーキテクチャ図を事例に |
| **FT-05** | 継続改善の仕組み → 長期契約の根拠 | レッドチーム結果を事例に |
| **FT-06** | 監査対応セット提案 → 差別化決定打 | ISO 42001 対応を事例に |
| **FT-07** | 具体的計画 → 予算確保の促進 | PoC 成果物を事例テンプレートに |

## 5. Walking Skeleton の戦略的意味

### 5.1 Walking Skeleton = 最小限の商談加速パス

チェックリスト内で定義された Walking Skeleton は、約150項目中の **約28項目** に集中する。これは各フィーチャーの「核心エピック」1つずつで構成される。

```
FT-01       FT-02       FT-03       FT-04       FT-05       FT-06       FT-07
EP-01-003   EP-02-001   EP-03-001   EP-04-002   EP-05-002   EP-06-002   EP-07-001
3層定義     In/Out      PoC成功     多層防御     テスト       監査         スプリント
合意        明確化      状態        設計        シナリオ     パッケージ    計画
```

### 5.2 Walking Skeleton の活用

1. **初回ヒアリングの「勝ちパターン」**: 28項目を合意できれば、PoC 提案の骨格が固まる
2. **営業ツールとしての活用**: チェックリストを顧客と共有し、「一緒に埋めていく」体験を提供
3. **事例化のテンプレート**: Walking Skeleton の合意プロセス自体を方法論として事例化

## 6. リスクと対策

| リスク | 影響 | 対策 |
|:---|:---|:---|
| ファーストカスタマーが見つからない | 事例化が進まず、R2 に移行できない | 既存顧客からの横展開、パートナー経由での紹介を並行推進 |
| PoC が期待値に達しない | 信頼失墜、事例化不可 | 赤線（最低限達成ライン）を Sprint 0 で合意し、早期に Go/No-Go 判定 |
| 4プラットフォーム統合の複雑さ | PoC 期間の超過、コスト増 | R1 では2プラットフォームに絞り、段階的に拡張 |
| 日本語対応の精度不足 | ガードレールの実用性低下 | PoC 実測項目として計画し、事前に期待値を調整 |
| 競合の出現 | 差別化の希薄化 | 監査証跡 + 構造化アプローチの先行者優位を確立 |
